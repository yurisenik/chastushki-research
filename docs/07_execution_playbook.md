# MVP Execution Playbook

Этот документ фиксирует операционный режим реализации MVP: субагентная декомпозиция, TDD-first и обязательное документирование изменений.

## Subagent workflow

- Один orchestrator управляет порядком работ и финальной интеграцией.
- Независимые треки исследуются параллельно, изменения интегрируются последовательно.
- На каждый трек задается контракт:
  - цель;
  - входные файлы;
  - список действий (3-7 пунктов);
  - что нельзя менять;
  - Definition of Done;
  - формат результата (`Что сделано`, `Какие файлы`, `Тесты`, `Риски`, `Changes made`).

## TDD-first

Для каждого инкремента:
1. `Red`: написать или расширить тест, получить падение.
2. `Green`: сделать минимальную реализацию для прохождения теста.
3. `Refactor`: улучшить код без изменения поведения и повторно прогнать тесты.

Минимальный пакет проверок на каждый инкремент:
- backend: `pytest -q`, `ruff check ...`;
- frontend: `npm run test`, `npm run lint`.

## Change records

После каждого инкремента обязательно публиковать блок `Changes made`:
- что изменено;
- зачем;
- где (файлы/области);
- какие проверки выполнены и их результат.
