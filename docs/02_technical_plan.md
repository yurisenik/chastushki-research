# Технический план сервиса генерации русских частушек

Обновлено: 2026-04-14

## 1. Цель сервиса

Сервис принимает тему, тональность, уровень «озорства», желаемый мотив и тип аранжировки, а на выходе даёт:
1. текст частушки на русском языке,
2. проверенную по форме версию текста,
3. аудио: либо выразительное чтение, либо пение под фиксированный мотив,
4. при необходимости несколько вариантов аранжировки.

Ниже я исхожу из практичной цели: сначала сделать управляемый MVP, а затем версию с более качественным вокалом и музыкальной подачей.

---

## 2. Что считать «хорошей» частушкой технически

Для генератора полезно формализовать жанр как набор ограничений:

- **4 строки**,
- короткая, легко пропеваемая форма,
- разговорная, народная, слегка шутливая интонация,
- тема должна считываться сразу,
- рифма чаще всего простая и явная,
- ритм должен быть достаточно ровным, чтобы текст ложился на один и тот же мотив,
- без запрещённого контента, оскорблений по защищённым признакам, призывов к насилию и незаконного сексуализированного/экстремального материала.

Для продакшена не стоит пытаться «доверить всё одной модели». Лучше работает **pipeline с генерацией, проверкой, оценкой и перегенерацией**.

---

## 3. Pipeline генерации текста частушек

## 3.1. Входные параметры

Рекомендую сделать такой контракт API:

```json
{
  "topic": "про дачу и весну",
  "style": "народная озорная",
  "spice_level": 1,
  "target_meter": {
    "lines": 4,
    "syllables_per_line": [8, 8, 8, 8],
    "stress_profile": "loose"
  },
  "rhyme_scheme": "AABB",
  "audience": "general",
  "melody_id": "chastushka_01",
  "arrangements": ["accordion", "balalaika", "band"]
}
```

`spice_level` лучше держать в безопасном диапазоне, например 0..2, где 0 это совсем семейный вариант.

## 3.2. Этапы текстового pipeline

### Этап A. Нормализация запроса

- привести тему к краткому тезису,
- выделить ключевые слова,
- определить допустимый стиль,
- применить policy-фильтры до генерации,
- сформировать скрытый `creative brief` для модели.

Пример внутреннего брифа:

- жанр: русская частушка,
- 4 строки,
- простая лексика,
- без архаизации «через силу»,
- иронично, но без токсичности,
- финальная строка должна содержать punchline,
- каждая строка около 7..9 слогов,
- рифма AABB или ABCB,
- избегать канцелярита, англицизмов и слишком длинных слов.

### Этап B. Черновая генерация нескольких кандидатов

Генерировать не 1, а **8-32 кандидата** с разными seed/temperature.

Подходы:
- **LLM-only**: модель сразу пишет готовые четверостишия по строгому шаблону.
- **LLM + constrained decoding**: модель генерирует по строкам, а на каждом шаге работают фильтры длины и рифмы.
- **Two-pass**: сначала план, затем финальный текст.

На практике для жанровой короткой формы хорошо работает именно **two-pass**:
1. модель делает план: тема, образ, шутка, рифмующиеся окончания,
2. затем пишет готовые 4 строки по плану.

### Этап C. Автоматическая проверка формы

Для каждого кандидата нужно считать формальные метрики:

1. **Количество строк = 4**.
2. **Количество слогов в строке**. Для частушечного мотива обычно достаточно окна, например 7..9 или 8..10 слогов.
3. **Ударения и удобопроизносимость**. Для русского это важно, иначе текст плохо поётся.
4. **Рифма**. Проверять не только последние буквы, а ударную часть финального слова.
5. **Стилевая близость**. Простота, разговорность, наличие шутки, отсутствие книжности.
6. **Тематическое попадание**.
7. **Safety/moderation**.

### Этап D. Ранжирование кандидатов

Итоговый score можно считать как взвешенную сумму:

```text
score = 0.25 * topic_fit
      + 0.20 * rhythm_score
      + 0.20 * rhyme_score
      + 0.15 * singability
      + 0.10 * style_score
      + 0.10 * safety_score
```

Кандидаты ниже порога отбрасываются. Если не прошёл никто, запускается перегенерация с подсказкой, какие ошибки были у предыдущих попыток.

### Этап E. Локальная правка лучшего кандидата

После отбора полезен отдельный этап `repair`:

- укоротить слишком длинную строку,
- заменить слово на более певучее,
- усилить концовку,
- подтянуть рифму,
- убрать сложные сочетания согласных.

Это может делать либо тот же LLM по жёсткому prompt, либо набор детерминированных правил + LLM.

## 4. Как технически контролировать размер строки, рифму и стиль

## 4.1. Размер строки

Минимально рабочий подход:
- считать слоги по гласным,
- задавать допустимое окно,
- штрафовать строки вне окна,
- иметь словарь замен для сокращения/удлинения строки.

Для русского лучше добавить:
- токенизацию,
- лемматизацию,
- оценку ударения,
- простую эвристику «плохих мест для пения».

Полезные инструменты:
- **pymorphy2** для морфологии,
- **русские ударения / stress-модели**: например `stressrnn` или аналогичный модуль расстановки ударений,
- **rupo** для русской поэзии: разметка ударений, слогов, рифмы и ритма.

Почему это важно: даже если LLM «на глаз» пишет короткие строки, без постпроверки русские тексты регулярно дают сбой в ударении и ломают напев.

## 4.2. Рифма

Рекомендую не ограничиваться суффиксом слова. Надёжнее такой алгоритм:

1. взять последнее содержательное слово строки,
2. расставить ударение,
3. выделить «рифмующее ядро» от ударной гласной до конца,
4. сравнить с нужной строкой,
5. дополнительно учитывать фонетическое сходство.

Пример:
- «малина» и «долина» рифмуются лучше, чем слова с одинаковыми последними двумя буквами, но другим ударением.

Практически:
- на MVP можно использовать словарь ударений + эвристики,
- в качественной версии лучше сделать свой `rhyme scorer` на основе фонемной записи слова.

## 4.3. Стиль

Стиль лучше контролировать комбинацией:

- prompt-инструкций,
- few-shot примеров,
- стилевого ranker-а,
- «чёрного списка» нежелательных признаков.

Что штрафовать:
- современный офисный жаргон, если он не нужен темой,
- чрезмерную литературность,
- длинные причастные конструкции,
- англицизмы,
- сложные многосложные слова,
- абстрактные фразы без punchline.

Дополнительно полезно обучить маленький бинарный классификатор `is_chastushka_style` на разметке «похоже / не похоже».

## 4.4. Культурные ограничения

Это отдельный слой, не просто moderation API.

Нужно различать:
- допустимую народную колкость,
- сексуальный или грубый юмор в мягкой форме,
- недопустимые оскорбления по защищённым признакам,
- политическую агитацию,
- клевету про реальных людей,
- контент, который выглядит как «народный», но фактически воспроизводит вредные стереотипы.

Практически нужен **policy layer**:
- pre-filter темы,
- post-filter текста,
- режимы `family_safe`, `light_folk`, `adult_but_safe`.

Для B2C-продукта я бы рекомендовал по умолчанию только `family_safe` и `light_folk`.

## 5. Рекомендуемая архитектура текстового модуля

```text
User request
  -> Input normalizer
  -> Policy filter
  -> Prompt builder
  -> LLM candidate generator (N samples)
  -> Prosody/rhyme analyzer
  -> Style + topic ranker
  -> Safety filter
  -> Repair pass
  -> Final scorer
  -> Best text + metadata
```

Метаданные стоит сохранять:
- слоги по строкам,
- последние слова,
- тип рифмы,
- оценки по каждому критерию,
- причина отклонения кандидатов.

Это сильно упростит отладку.

## 6. Варианты синтеза пения и озвучивания под фиксированный мотив

Здесь есть два разных продукта.

### Вариант 1. Выразительное чтение, не пение

Самый простой и надёжный путь для MVP:
- генерировать частушку,
- озвучивать её как ритмизованное чтение,
- подкладывать заранее подготовленный аккомпанемент.

Плюсы:
- быстро внедряется,
- предсказуемое качество,
- меньше проблем с интонацией,
- ниже требования к датасету.

Минус:
- это не настоящее пение.

### Вариант 2. Настоящее пение по фиксированной мелодии

Нужна система, которая принимает:
- текст,
- последовательность нот и длительностей,
- темп,
- возможно фонемную разметку,
- стиль вокала.

Дальше есть несколько подходов.

## 6.1. Подход A. Score-conditioned singing synthesis

Это наиболее правильный путь для качественной версии.

Pipeline:
1. подготовить **фиксированную мелодию** как MIDI/notes,
2. разбить текст на слоги/фонемы,
3. выровнять слоги по нотам,
4. подать `lyrics + notes + durations` в модель певческого синтеза,
5. свести вокал с аранжировкой.

Подходящие инструменты:
- **DiffSinger**,
- совместимые вокальные пайплайны вокруг **OpenSVI** / **OpenCpop**-style экосистем,
- **PaddleSpeech Singing Voice Synthesis** как ориентир по архитектуре и компонентам.

Плюсы:
- точное следование мотиву,
- управляемость по нотам,
- можно сделать несколько голосов.

Минусы:
- сложнее в проде,
- нужен специализированный вокальный стек,
- русский язык поддерживается хуже, чем китайский/английский, часто нужен свой адаптационный слой.

## 6.2. Подход B. Речь + мелодизация / pitch correction

Pipeline:
1. TTS делает выразительное чтение,
2. forced alignment разбивает аудио по фонемам/слогам,
3. pitch-shifting / time-stretch подтягивает фразы к нотам,
4. сверху добавляются декоративные вокальные эффекты.

Инструменты:
- TTS: **OpenAI gpt-4o-mini-tts** или **gpt-4o-transcribe/tts family** для озвучки, либо локальный **Silero TTS / Coqui TTS**,
- alignment: **Montreal Forced Aligner** или аналоги,
- коррекция высоты: DAW-плагины, Rubber Band, Melodyne-подобный стек,
- постобработка: EQ, reverb, compression.

Плюсы:
- проще, чем полноценный singing model,
- можно быстро получить «полупение» или речитатив под частушечный мотив.

Минусы:
- натуральность хуже,
- на длинных гласных и скачках мелодии слышны артефакты,
- сложно добиться настоящего народного вокала.

Для MVP этот путь часто разумнее.

## 6.3. Подход C. Voice conversion поверх эталонного исполнения

Pipeline:
1. базовый певец или студийный исполнитель поёт частушку,
2. затем применяется voice conversion / timbre transfer,
3. на выходе получаем разные «образы» голоса.

Плюсы:
- высокое качество при наличии хорошего оригинала,
- легко получить несколько тембров.

Минусы:
- самые большие правовые и этические риски,
- нужны права на исходные голоса,
- нельзя без разрешения делать стилизацию под узнаваемых артистов.

Для публичного сервиса я бы использовал этот вариант только с полностью лицензированными голосами и явным согласием исполнителей.

## 7. Как привязать текст к фиксированному мотиву

Это ключевой инженерный кусок.

## 7.1. Представление мотива

Для каждого мотива хранить:
- MIDI,
- список нот,
- длительности,
- опорные доли,
- шаблон «сколько слогов ложится в строку».

Пример:

```json
{
  "melody_id": "chastushka_01",
  "meter": "2/4",
  "tempo": 132,
  "lines": 4,
  "line_templates": [
    {"target_syllables": 8, "notes": 8},
    {"target_syllables": 8, "notes": 8},
    {"target_syllables": 8, "notes": 8},
    {"target_syllables": 8, "notes": 8}
  ]
}
```

## 7.2. Алгоритм выравнивания

1. Посчитать слоги в строке.
2. Если слогов больше шаблона, попытаться сократить строку через repair.
3. Если меньше, разрешить:
   - распев одной гласной на две ноты,
   - междометие или короткую вставку,
   - мелизму без новой лексемы.
4. Перевести строку в фонемы.
5. Назначить длительности фонемам внутри ноты.
6. Отдать это в singing/TTS backend.

Лучше всего работает не «свободная генерация текста», а генерация **сразу под конкретный слоговой шаблон мотива**.

## 8. Аранжировки

Для частушек нет смысла в начале делать генеративную аранжировку с нуля. Надёжнее иметь библиотеку готовых стилей.

## 8.1. Для MVP

Сделать 3-5 заранее собранных аранжировок:
- **гармонь / баян**,
- **балалайка + хлопки**,
- **деревенский ансамбль**,
- **ярмарочный быстрый вариант**,
- **минималистичный фон под чтение**.

Технически это может быть:
- MIDI + soundfonts,
- лупы и one-shot-сэмплы,
- DAW-шаблоны, которые рендерятся серверно.

Плюс такого подхода в том, что пользователь получает стабильный стиль и не зависит от музыкальной генерации каждый раз.

## 8.2. Для более качественной версии

Варианты:

1. **Rule-based arrangement engine**
   - один и тот же мотив,
   - меняются инструменты, заполнения, вступление, концовка.

2. **MIDI-first генерация**
   - генерируется или варьируется аккомпанемент в MIDI,
   - затем рендер через виртуальные инструменты.

3. **Генеративная музыка**
   - например MusicGen-подобные модели,
   - но у них хуже управляемость по точной структуре и больше вопросов по лицензиям/повторяемости.

Для сервиса частушек я рекомендую **не ставить музыку на полностью генеративные рельсы** на ранней стадии. Лучше иметь curated backing tracks.

## 9. Что использовать для MVP

## 9.1. Текст

### Вариант MVP

- **LLM**: современная сильная instruction-модель через API, которая хорошо держит русский.
- Постобработка:
  - `pymorphy2`,
  - словарь ударений / `stressrnn`,
  - `rupo` или собственный анализатор слогов и рифмы,
  - moderation layer.

Практический паттерн:
- генерировать 16 кандидатов,
- фильтровать по длине строки,
- считать рифму,
- отдельной моделью ранжировать,
- лучший вариант чинить repair-pass-ом.

### Почему не только one-shot prompt

Потому что даже очень хорошая LLM:
- иногда пишет слишком длинные строки,
- теряет ударение,
- даёт слабую концовку,
- может делать «стих вообще», а не частушку.

Нужна структура вокруг модели.

## 9.2. Аудио для MVP

Рекомендую такой стек:

### Самый практичный MVP

- **Озвучивание**: TTS, не пение.
- **Ритмизация**: паузы и темп под мотив.
- **Музыка**: 3-5 заранее подготовленных аранжировок.
- **Сведение**: ffmpeg + лёгкая постобработка.

Подходящие инструменты:
- облачный TTS: **OpenAI audio/TTS API**,
- локальный TTS: **Silero TTS** или **Coqui TTS**,
- сборка аудио: **ffmpeg**, **sox**, DAW batch render.

### Если очень хочется «пение» уже в MVP

Лучше сделать **spoken-sing**:
- TTS с более протяжными гласными,
- лёгкая коррекция высоты на ключевых слогах,
- фон с гармоникой/балалайкой.

Это будет убедительнее, чем слабый нейро-вокал без датасета.

## 10. Что использовать для более качественной версии

## 10.1. Текстовый модуль

- LLM для генерации и repair,
- отдельный **prosody checker** для русского,
- отдельный **style ranker**,
- отдельный **rhyme scorer** на фонемах,
- dataset с разметкой «хорошая/плохая частушка» и «поётся/не поётся».

Дополнительно можно обучить маленькую reranker-модель на собственных логах и пользовательских лайках.

## 10.2. Вокал

Предпочтительная цель:
- **score-conditioned singing synthesis**,
- русский фонемизатор,
- привязка текста к нотам,
- несколько собственных лицензированных голосов.

Технологически:
- DiffSinger-подобный стек,
- фонемизация + forced alignment,
- вокальный vocoder,
- отдельный модуль сведения.

## 10.3. Музыка

- библиотека MIDI-паттернов и интро/аутро,
- серверный рендер через VST/сэмплеры,
- опционально генерация вариаций в MIDI, а не сырым аудио.

## 11. Риски

## 11.1. Риски качества

1. **LLM пишет стих, но не частушку**.
   - Решение: отдельный genre-ranker и strict template.

2. **Строки не ложатся на мотив**.
   - Решение: слоговый шаблон, repair, prosody scoring.

3. **Слабая рифма**.
   - Решение: фонемный rhyme scorer, генерация нескольких кандидатов.

4. **Пение звучит неестественно**.
   - Решение: в MVP не обещать полноценный вокал; использовать выразительное чтение или лицензированный студийный пайплайн.

5. **Русский фольклорный стиль получается карикатурным**.
   - Решение: аккуратный датасет, ручная редактура seed-примеров, human evaluation.

## 11.2. Риски прав

1. **Датасеты текстов**.
   - Нужно проверять, находятся ли тексты в public domain, по какой лицензии доступны цифровые сборники и можно ли использовать конкретную разметку.
   - Старые народные тексты как жанр не защищены, но конкретные современные издания, комментарии и оцифровки могут иметь права.

2. **Голоса**.
   - Нельзя имитировать узнаваемого живого исполнителя без разрешения.
   - Лучше использовать лицензированные синтетические голоса или контрактные студийные записи.

3. **Музыкальные аранжировки**.
   - Лучше писать свои backing tracks или использовать сэмплы/библиотеки с явной коммерческой лицензией.

4. **Open-source лицензии моделей**.
   - Нужно отдельно проверять коммерческое использование, требования к атрибуции и ограничения на веса/датасеты.

## 11.3. Safety-риски

1. Пользователь может просить:
   - оскорбительные частушки про конкретного человека,
   - политический троллинг,
   - sexual content,
   - уничижение групп.

2. Народный жанр провоцирует «пограничный юмор».
   - Нужны режимы безопасности и ясные продуктовые ограничения.

3. Голосовой output усиливает вред.
   - Для озвучки нужны те же policy-checks, что и для текста.

## 11.4. Риски по датасетам

1. **Мало хорошо размеченных русских частушек**.
2. **Шумные OCR-источники** в старых сборниках.
3. **Перекос в сторону устаревшей лексики**.
4. **Токсичные исторические образцы**.
5. **Певческие датасеты для русского** заметно слабее и меньше, чем для английского/китайского.

Поэтому я бы строил продукт не на большом дообучении «с нуля», а на:
- сильной базовой LLM,
- curated text set для few-shot/ranking,
- небольшом собственном наборе аудио для фирменных голосов.

## 12. Данные и разметка

## 12.1. Текстовый корпус

Нужны три набора:

1. **Seed corpus частушек**
   - только тексты с понятным правовым статусом,
   - очистка от OCR-ошибок,
   - пометки темы, тона, рифмы, слогов.

2. **Negative corpus**
   - плохие стихи,
   - тексты не в жанре,
   - небезопасные примеры.

3. **Evaluation set**
   - 300-1000 запросов с ручной оценкой.

## 12.2. Аудиоданные

Для качественной версии нужны:
- лицензированные записи одного или нескольких певцов,
- отдельные фонемные/слоговые выравнивания,
- записи на нескольких темпах,
- желательно и spoken, и sung варианты.

Если делать фирменный голос сервиса, лучше сразу получать:
- договор на синтез и производные модели,
- право коммерческого использования,
- разрешение на voice cloning / synthesis, если это вообще планируется.

## 13. Оценка качества

Нужно мерить не только «нравится / не нравится».

## 13.1. Текст

Автоматические метрики:
- доля текстов, прошедших форму с первого раза,
- среднее отклонение по слогам,
- rhyme pass rate,
- moderation reject rate,
- semantic relevance.

Человеческая оценка:
- похоже ли на частушку,
- смешно ли,
- легко ли пропеть,
- нет ли кринжа / канцелярита,
- хочется ли переслать другу.

## 13.2. Аудио

- разборчивость слов,
- попадание в ритм,
- естественность,
- «ощущается ли как частушка»,
- MOS по голосу.

## 15. Моя практическая рекомендация

Если цель, чтобы сервис реально заработал и не развалился по качеству, я бы выбрал такой курс:

### Рекомендованный MVP

- **Текст**: сильная LLM + формальные проверки + repair.
- **Аудио**: не настоящее пение, а **ритмизованное озвучивание**.
- **Музыка**: набор заранее сделанных аранжировок.
- **Безопасность**: жёсткий safe-by-default режим.

Это даст:
- быстрый запуск,
- управляемое качество,
- меньше правовых рисков,
- предсказуемую себестоимость.

### Рекомендованная качественная версия

- тот же текстовый pipeline,
- плюс **score-conditioned singing synthesis**,
- плюс собственные лицензированные певческие голоса,
- плюс MIDI-first аранжировки.

Именно такой путь лучше всего балансирует качество, контроль и юридическую чистоту.

## 18. Вывод

Лучший инженерный компромисс такой:

1. **Текст генерировать через LLM, но обязательно через формальный pipeline**: multi-candidate, проверка слогов, рифмы, стиля и safety.
2. **Для MVP не делать ставку на нейропение**. Делать хорошее ритмизованное озвучивание под готовый частушечный backing track.
3. **Для качественной версии** переходить к score-conditioned singing synthesis с собственными лицензированными голосами и MIDI-first управлением мотивом.
4. **Главные риски** не только в качестве, но и в правах на голос, датасеты и в культурно-чувствительном контенте. Их надо проектировать в систему с первого дня.

Если нужно, следующим шагом я бы сделал ещё два артефакта:
1. схему API/таблиц для этого сервиса,
2. пример prompt set и scoring rules для текстового генератора.
